# PF 사례 검증 데이터 (data/pf-cases)

이 폴더는 AI(이 앱)의 분석 결과와 **실제 PF 사업 결과**를 나중에 비교·검증하기 위한
데이터 구조입니다. 지금은 구조와 더미 데이터만 있고, 비교 로직/UI는 아직 없습니다
(향후 별도 기능으로 추가 예정).

## 파일 구성

- `index.json` — 이 폴더에 있는 사례 파일 목록(향후 로더가 이 파일만 보고 전체 사례를
  동적 glob 없이 나열할 수 있게 하기 위함)
- `case-XXX.json` — 사례 1건. 스키마는 아래 참고

## 사례 스키마 (`case-XXX.json`)

```jsonc
{
  "id": "case-001",                 // 파일명과 동일하게 유지
  "caseName": "사람이 읽을 사례명",
  "status": "dummy",                // "dummy"(더미) | "verified"(실제 사례로 확인됨)
  "source": "출처 — 실제 사례로 교체 시 여기에 근거(계약서/공시자료 등) 명시",
  "input": {
    // src/components/PFReportMVP.jsx의 `form` state와 동일한 필드 구조.
    // 이렇게 맞춰두면 향후 로더가 이 값을 그대로 runAnalysis()에 넣어
    // "AI가 이 입력값으로 냈을 결과"를 재현할 수 있습니다.
    "address": "", "area": "", "zone": "", "projectType": "", "lender": "",
    "developerTrack": "", "contractorGrade": "", "locationTier": "",
    "supplyCompetition": "", "creditEnhancement": "", "permitStage": "",
    "expectedSaleRate": "", "equityRatio": "", "interestRate": "",

  "actual": {
    // 실제 사업 진행 결과(더미 사례는 값을 비워두거나 가정치로 채움)
    "totalCost": null,       // 실제 총사업비(만원)

  "aiResult": null           // 이 앱으로 분석을 돌린 결과 캐시(등급/총점 등). 향후 비교
```

   애초에 실제 사례보다 더미로 남기는 게 더 적합한 유형)만 더미로 유지. 총 10건(더미 1 + 실제 9)

## 주의

- `case-005`를 제외한 나머지 9개 사례는 **실제 사례**입니다(`status: "verified"`). `case-005`만
  더미 데이터입니다(`status: "dummy"`, `source`에 명시) — 실제 사업지·시행사 정보가 아니므로
  그대로 실서비스 근거로 쓰면 안 됩니다.
- `case-001`(`status: "verified"`)은 서울시 정비사업 정보몽땅(공공포털)의 대지면적·세대수와
  언론보도(청약경쟁률·분양가·입주일)가 근거인 실제 사례(래미안 원베일리)입니다. 용도지역과
  PF 금융조건(금리·자기자본비율 등)은 비공개라 앱의 기본 미정값을 그대로 썼습니다.
- `case-002`(`status: "verified"`)는 언론보도가 근거인 실제 사례(힐스테이트 동탄역 센트릭)로,
  대지면적·규모·청약 경쟁률(29.8:1)·완판 시점은 실제 수치입니다. 용도지역·시행사명·PF 대출조건은
  비공개라 앱의 기본 미정값을 썼습니다.
- `case-003`(`status: "verified"`)은 지역신문 보도가 근거인 실제 사례(통영 더유엘 윈썸)로,
  주소·대지면적·규모·시공사 부도 경위·HUG 처리 방식(계속사업)은 실제 수치입니다. 용도지역·
  PF 대출조건·정확한 분양률은 비공개라 앱의 기본 미정값을 썼습니다.
- `case-004`(`status: "verified"`)는 주소·대지면적·연면적·타임라인·공매 이력은 언론 보도(제주의소리)
  기준 실제 수치이지만, 정확한 지번·용도지역·시행사명·PF 금융조건(금리·자기자본비율 등)은 비공개
  정보라 앱의 기본 미정값을 그대로 썼습니다 — `source` 필드에 어떤 값이 실제고 어떤 값이 미정
  기본값인지 명시되어 있으니 참고하세요.
- `case-006`(`status: "verified"`)은 대구 골든프라자(복현SKY) — 1989년 착공 후 36년 미준공,
  HUG 공매로 감정가의 51%에 낙찰된 국내 최장기급 미준공 사례. 서울경제 보도 + HUG 공매정보

### 검증 결과로 발견하고 수정한 채점모델 한계 (v1.3.0 → v1.4.0, 해결됨)

`case-004`·`case-006`·`case-007`·`case-008`·`case-009`·`case-010`(전부 실제 결과 `default`)을
처음 반영했을 때, 6건 중 5건이 AI등급 `BBB`대(양호)로 나와 실제 결과와 **"불일치"** 판정이었습니다.

원인: `src/lib/scoring/index.js`의 "사업안정성" 하드게이트가 인허가·시행사·시공사 **3개 전부**
위험이어야 발동하도록 되어 있었는데, 실제 부도 사례는 대부분 착공 이후(인허가 "진행 중" 단계)에
발생해 인허가 항목은 위험으로 안 잡히고, 시행사·시공사 2개만 위험이면 게이트가 안 걸렸습니다.
(형제 게이트인 "금융" 게이트는 애초에 4개 중 2개 이상이면 발동하도록 되어 있어 기준이 비대칭이었음.)

**수정 1(v1.4.0)**: 사업안정성 게이트도 "2개 이상 위험"으로 금융 게이트와 기준을 통일. 수정 후
재검증하니 6건 전부 "일치"로 정확히 잡힙니다.

**수정 2(v1.4.1)**: "재무조건 미정 시 안전해 보이는 기본값 때문에 리스크를 놓치는" 구조적 위험도
후속 조치. 자기자본비율의 기본값(20%)이 하필 "우수" 컷오프(20% 이상)와 정확히 같아서, 미입력
상태로 두면 항상 만점을 받고 있었습니다(시행사·시공사처럼 "미정(정보 없음)"이면 위험 처리되는
정성 항목들과 달리, 이 정량 항목엔 그런 처리가 없었음). `equityRatioIsDefault` 플래그를 추가해
미입력이면 이 항목도 다른 정성 항목과 동일하게 위험 처리하도록 통일(`src/lib/scoring/index.js`).
LTV·DSCR·금융비용부담은 여러 입력값이 얽힌 계산값이라 같은 방식 적용이 애매해 이번엔 손대지
않았고, 게이트 자체를 다시 트리거하지 않아도 이미 6/6 사례가 정확히 잡히는 걸 확인했습니다.
